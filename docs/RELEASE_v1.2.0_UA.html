1. Загальний огляд

Реліз v1.2.0 — це масштабне оновлення інфраструктури та операційної зрілості платформи FICE Selection Committee. Основний фокус: надійність збірок, коректність контрактів, розширення функціональності та спостережуваність системи.

Цей реліз приносить ключові покращення в кількох напрямках:

  • Багатоетапні Docker-збірки — триетапний процес побудови образів, що забезпечує відтворюваність та мінімальний розмір runtime-образу.
  • Виправлення Auth API — зміна HTTP-методу getCurrentUser з POST на GET для семантичної коректності.
  • Новий Telegram Bot сервіс — управління feature-прапорцями через Telegram з підтримкою webhook, inline-клавіатур та i18n.
  • Розширений CI/CD — нова матриця збірок, сканування вразливостей Trivy, семантичне тегування Docker-образів.
  • Моніторинг Grafana — 4 попередньо налаштовані дашборди для операційного моніторингу.
  • Інженерна документація — CLAUDE.md та 10 довідкових файлів як стандарт інженерної роботи.

3. Що саме змінено

3.1 Docker: Перехід на багатоетапні збірки ЗМІНЕНО

Усі сервіси переведено на триетапну Docker-збірку, що забезпечує ізоляцію етапів, мінімальний runtime-образ та правильний порядок побудови спільних бібліотек.

1
lib-builder
eclipse-temurin:21-jdk-alpine
Побудова спільних бібліотек (publishToMavenLocal)
2
service-builder
Збірка JAR сервісу з використанням бібліотек з етапу 1
3
runtime
eclipse-temurin:21-jre-alpine
Мінімальний образ + HEALTHCHECK

Ключові рішення:

  • Базовий образ змінено з gradle:jdk21-ubi-minimal на eclipse-temurin:21-jdk-alpine (збірка) та eclipse-temurin:21-jre-alpine (runtime)
  • Стандартизований користувач spring з UID/GID 10000/10001 у всіх сервісах
  • Додано server/.dockerignore для оптимізації контексту збірки
  • Вбудований HEALTHCHECK для кожного контейнера

Приклад структури Dockerfile:

# Етап 1: Побудова спільних бібліотек
FROM eclipse-temurin:21-jdk-alpine AS lib-builder
WORKDIR /workspace
COPY libs/ libs/
COPY buildSrc/ buildSrc/
RUN ./gradlew :libs:sc-auth-starter:publishToMavenLocal

# Етап 2: Збірка сервісу
FROM eclipse-temurin:21-jdk-alpine AS service-builder
COPY --from=lib-builder /root/.m2 /root/.m2
RUN ./gradlew :services:my-service:bootJar

# Етап 3: Runtime
FROM eclipse-temurin:21-jre-alpine
RUN addgroup -g 10001 spring && adduser -u 10000 -G spring spring
COPY --from=service-builder /workspace/build/libs/*.jar app.jar
HEALTHCHECK CMD wget -qO- http://localhost:8080/actuator/health || exit 1

3.2 Авторизація: POST → GET для getCurrentUser ЗМІНЕНО

Ендпоінт отримання поточного користувача змінено з POST на GET для дотримання REST-семантики — операція читання даних не повинна використовувати метод, що передбачає створення ресурсу.

КомпонентЗміна
IdentityServiceClient@PostMapping → @GetMapping для /api/v1/auth/user
ProviderContractITОновлено: тепер використовує потік magic link автентифікації
IdentityClientContractIT (documents)Контрактний тест оновлено відповідно до нового методу

Зверніть увагу: Це ламаюча зміна (breaking change) для будь-яких зовнішніх клієнтів, що використовують POST /api/v1/auth/user. Усі внутрішні Feign-клієнти та контрактні тести оновлено.

3.3 Інфраструктура: Docker Compose ПОКРАЩЕНО

Значні зміни у конфігурації Docker Compose для підвищення надійності та підтримки нових сервісів:

  • Залежності на основі здоров'я: service_started → service_healthy для Redis та RabbitMQ — сервіси чекають на фактичну готовність інфраструктури
  • Контекст збірки: змінено на кореневу директорію server/ для підтримки багатоетапних збірок з доступом до спільних бібліотек
  • JWT секрети: volume mounts для identity-service
  • Rate limiting: конфігурація обмеження частоти для magic link
  • Telegram Bot: новий сервіс telegram-bot-service + ngrok + webhook-init у профілі telegram
  • Іменування змінних: виправлено RABBIT_* → RABBITMQ_*
  • Health check шлюзу: виправлено порт з 8080 → 8079 (management port)
  • .env.example: створено шаблон з документацією усіх змінних оточення

Архітектурні принципи v1.2.0

  • Усі сервіси використовують багатоетапні Docker-збірки з попередньою публікацією спільних бібліотек
  • Координація залежностей на основі здоров'я (service_healthy) для всіх сервісів
  • Magic link автентифікація — без паролів, лише email-посилання
  • Feature flags керуються через environment-service та доступні через Telegram-бот
  • Grafana дашборди для операційного моніторингу провізіонуються автоматично
  • Кожен сервіс має HEALTHCHECK у Dockerfile та /actuator/health ендпоінт

6. Як запускати

Підготовка середовища

# Скопіюйте шаблон змінних оточення та заповніть значення
cp infra/.env.example infra/.env

Запуск інфраструктури

# Запуск PostgreSQL, Redis, RabbitMQ, MinIO, Mailpit, Zipkin, Prometheus, Grafana
cd infra && docker compose up -d

Запуск сервісів

# Запуск усіх основних мікросервісів
docker compose -f docker-compose.services.yml up -d

Запуск з Telegram Bot (опціонально)

# Включає telegram-bot-service + ngrok + webhook-init
docker compose -f docker-compose.services.yml --profile telegram up -d

Передумови: Docker та Docker Compose мають бути встановлені. Усі сервіси запускаються у мережі sc-net. Перша збірка може тривати 5-10 хвилин через завантаження залежностей.

7. Як деплоїти

ТригерДія
Push до mainЗапускає Docker CI — збірка та публікація образів із змінених сервісів
Тег v*.*.*Повна перезбірка усіх сервісів з тегуванням за semver
Публікація образівghcr.io/fice-selection-committee/sc-*
Сканування TrivyВиконується на кожній збірці, результати у GitHub Security (SARIF)

Процес деплою:

  1. Зміни проходять code review та CI-перевірки у pull request
  2. Після merge до main автоматично збираються Docker-образи змінених сервісів
  3. Trivy сканує кожен образ на вразливості та завантажує SARIF-звіт
  4. Для релізу створюється тег v*.*.*, що тригерить повну перезбірку
  5. Семантичні теги (latest, v1.2.0, v1.2, v1) застосовуються до образів

11. Висновки та подальші кроки

Ключові досягнення v1.2.0

  • Багатоетапні Docker-збірки зменшують розмір образів та покращують відтворюваність збірок. Виключення JDK та Gradle з runtime-образу значно зменшує поверхню атаки.
  • Залежності на основі здоров'я запобігають стартовим гонкам (race conditions) — сервіси не починають роботу, поки інфраструктура не підтвердить свою готовність.
  • Виправлення Auth API приводить контракт у відповідність до REST-семантики, що спрощує інтеграцію для зовнішніх клієнтів.
  • Telegram Bot додає зручний операційний інтерфейс для управління feature-прапорцями без необхідності доступу до адмін-панелі.
  • Grafana дашборди забезпечують миттєву спостережуваність системи з першого запуску.
  • Trivy у CI встановлює безперервний контроль вразливостей як частину процесу збірки.
  • Інженерна документація формалізує стандарти роботи, що підвищує передбачуваність та якість змін.
