Частина А

Нетехнічний огляд

Загальний опис платформи, нові можливості для користувачів, хронологія розвитку та плани на майбутнє — без технічної термінології.

А.1 Вступ

Цей документ описує еволюцію платформи FICE Selection Committee від створення до версії 1.2.0. Він розділений на дві частини: Частина А містить загальний огляд зрозумілою мовою без технічних деталей, Частина Б — повну технічну документацію для інженерної команди.

Платформа розроблялася протягом 6 днів (25–30 березня 2026 р.) і пройшла шлях від порожнього проєкту до повнофункціональної системи з автоматизованою збіркою, моніторингом, тестуванням та інженерними стандартами.

А.2 Загальний огляд платформи

FICE Selection Committee — платформа для управління процесом вступу до навчальних закладів. Система автоматизує ключові процеси приймальної комісії та забезпечує зручну взаємодію з абітурієнтами.

Основні функції платформи:

  • Автентифікація користувачів — безпечний вхід через одноразові email-посилання без необхідності запам'ятовувати паролі.
  • Управління документами — завантаження, зберігання та обробка документів абітурієнтів.
  • Обробка заявок на вступ — прийом та опрацювання заяв абітурієнтів.
  • Сповіщення — автоматичне інформування користувачів через email.
  • Управління функціональністю — можливість вмикати та вимикати окремі функції системи через месенджер, без необхідності зупиняти роботу платформи.
  • Моніторинг — автоматичне відстеження стану всіх компонентів системи з візуальними панелями.

А.3 Що нового у кожній версії

А.3.1 Версія 1.0.0 (25–26 березня): Фундамент

Перша версія заклала основу проєкту. Розрізнені компоненти системи, що існували окремо, були об'єднані в єдиний проєкт із спільною структурою. Створено перевикористовувані компоненти, щоб уникнути дублювання роботи між різними частинами системи. Налаштовано механізм публікації готових компонентів для подальшого використання.

А.3.2 Версія 1.1.0 (26–28 березня): Інфраструктура

Друга версія додала операційну інфраструктуру. З'явилося автоматичне тестування, що перевіряє коректність роботи системи при кожній зміні. Створено панелі моніторингу для відстеження здоров'я системи у реальному часі. Налаштовано повне локальне середовище розробки, де весь стек можна запустити однією командою. Впроваджено автоматичний конвеєр збірки та публікації.

А.3.3 Версія 1.2.0 (30 березня): Зрілість

Третя версія сфокусована на зрілості та безпеці. Додано автоматичне сканування на вразливості при кожній збірці. Створено бот у месенджері Telegram для оперативного управління системою. Повністю перероблено зовнішній вигляд платформи з новою системою входу через email-посилання (без паролів). Додано юридичні сторінки. Покращено процес контейнеризації для зменшення розміру та підвищення безпеки.

А.4 Нові можливості для користувачів

  • Вхід через email-посилання — замість запам'ятовування пароля користувач отримує одноразове посилання на електронну пошту. Один клік — і ви в системі.
  • Оновлений дизайн головної сторінки — нова панельна розкладка з поділом екрану на інформаційну та функціональну частини.
  • Триетапний онбордінг — покроковий процес реєстрації з обов'язковим ознайомленням із юридичними документами (правилами використання та угодою про конфіденційність).
  • Панель управління (Dashboard) — персональний кабінет користувача з доступом до основних функцій.
  • Юридичні сторінки — правила використання платформи та угода про конфіденційність, доступні для перегляду у будь-який час.
  • Управління функціональністю через Telegram — адміністратори можуть вмикати та вимикати функції платформи безпосередньо з месенджера Telegram.
  • Покращена доступність (WCAG) — інтерфейс адаптовано для людей з обмеженими можливостями: навігація клавіатурою, підтримка програм зчитування з екрану, покращений контраст кольорів.

Б.6 Версія 1.1.0 — Інфраструктура та тестування (26–28 березня 2026)

Б.6.1 Цілі та обсяг

Другий реліз додав операційну інфраструктуру: Docker Compose overlay для повного локального стеку, CI/CD конвеєр із 6 ітеративними hotfixes, контрактні та E2E тести, OpenAPI TypeScript-генерацію та Grafana дашборди.

Ключові цілі:

Архітектурні принципи

  • Bounded contexts: кожен сервіс володіє своїм доменом, базою даних та схемою. Міжсервісний доступ лише через Feign HTTP або RabbitMQ events.

  • Health-based залежності: service_healthy для усіх сервісів у Docker Compose.
  • Багатоетапні Docker-збірки: 3 етапи (lib-builder → service-builder → runtime) для мінімального образу.

  • Автоматичний моніторинг: Grafana дашборди провізіонуються при першому запуску.
  • Кожен сервіс має HEALTHCHECK: вбудований у Dockerfile + /actuator/health ендпоінт.
  • Фронтенд як окремий репозиторій: selection-committee-web (гілка develop) — Next.js 16, TypeScript, shadcn/ui, незалежний цикл розробки та деплою.

Б.12 Детальне порівняння «Було → Стало»

1. Структура репозиторію

Було: Окремі репозиторії для кожного сервісу
Стало: Gradle multi-module монорепозиторій
Як це працює: Усі мікросервіси, спільні бібліотеки та інфраструктурні конфігурації об'єднані у єдиний репозиторій з централізованою системою збірки. У директорії server/buildSrc/ визначено convention plugins: sc.java-conventions задає базові налаштування Java-проєкту, sc.spring-boot-service — конфігурацію Spring Boot сервісу, sc.testing-conventions — тестові залежності та налаштування, sc.jacoco-conventions — покриття коду. Централізоване управління залежностями через version catalogs забезпечує однакові версії бібліотек у всіх модулях. Єдина точка входу ./gradlew дозволяє збирати, тестувати та публікувати будь-який компонент.
Чому це важливо: Єдиний репозиторій унеможливлює версійну неузгодженість між сервісами, спрощує рефакторинг крос-сервісних змін та забезпечує атомарність релізів.

2. Docker-збірки

Було: Відсутні або одноетапні Dockerfile
Стало: Багатоетапні 3-стадійні збірки
Як це працює: Кожен Dockerfile містить три етапи. Етап 1 (lib-builder) на базі eclipse-temurin:21-jdk-alpine копіює спільні бібліотеки та виконує publishToMavenLocal, роблячи їх доступними для наступного етапу. Етап 2 (service-builder) копіює mavenLocal з попереднього етапу та збирає JAR-файл конкретного сервісу через bootJar. Етап 3 (runtime) використовує мінімальний eclipse-temurin:21-jre-alpine, створює non-root користувача spring (UID 10000, GID 10001), копіює лише зібраний JAR та встановлює вбудований HEALTHCHECK через wget до /actuator/health.
Чому це важливо: Фінальний образ містить лише JRE та JAR-файл, що значно зменшує поверхню атаки. Non-root користувач запобігає ескалації привілеїв всередині контейнера.

3. CI/CD конвеєр

Було: Відсутній
Стало: GitHub Actions з двома workflow
Як це працює: Workflow service-build-test.yml запускається при кожному Pull Request та виконує unit- та integration-тести для змінених сервісів. Workflow service-docker.yml активується при merge до main або створенні semver-тегу: він будує Docker-образи та публікує їх до GHCR. Динамічна матриця автоматично визначає, які сервіси змінилися (аналізуючи шляхи файлів у коміті), та збирає лише їх. Для кожного сервісу динамічно визначається контекст збірки та шлях до Dockerfile.
Чому це важливо: Автоматизація виключає людський фактор у процесі збірки та публікації. Динамічна матриця економить час, збираючи лише змінені сервіси.

4. Сканування вразливостей

Було: Відсутнє
Стало: Trivy у CI/CD конвеєрі
Як це працює: Trivy — open-source сканер від Aqua Security, що аналізує шари контейнерного образу на наявність відомих CVE (Common Vulnerabilities and Exposures). Він запускається як крок у service-docker.yml після кожної збірки образу. Trivy перевіряє всі пакети операційної системи та Java-залежності у фінальному образі. Результати генеруються у форматі SARIF (Static Analysis Results Interchange Format) та автоматично завантажуються до вкладки GitHub Security. Таймаут job встановлено на 30 хвилин.
Чому це важливо: Автоматичне сканування виявляє відомі вразливості ще до деплою, забезпечуючи безперервний контроль безпеки без ручного втручання.

5. Автентифікація

Було: Логін/пароль
Стало: Magic link (вхід через email-посилання)
Як це працює: Користувач вводить email-адресу на сторінці входу. Identity-service генерує JWT-токен з обмеженим терміном дії та вбудовує його у посилання. Notifications-service доставляє лист з посиланням через SMTP (у локальній розробці використовується Mailpit). Користувач натискає посилання, токен валідується на сервері, встановлюється сесійний cookie. Axios-інтерсептори на фронтенді автоматично оновлюють токен при його закінченні. Rate limiting налаштовано для ендпоінту magic link для захисту від зловживань.
Чому це важливо: Magic link виключає ризик компрометації паролів, фішингу через підроблені форми входу та використання слабких паролів користувачами.

6. Auth API семантика

Було: POST /api/v1/auth/user для getCurrentUser
Стало: GET /api/v1/auth/user для getCurrentUser
Як це працює: HTTP-метод POST семантично означає "створити ресурс", тоді як операція getCurrentUser лише зчитує дані. Виправлено @PostMapping на @GetMapping у Feign-інтерфейсі IdentityServiceClient. Оновлено ProviderContractIT для відповідності потоку magic link автентифікації. Оновлено IdentityClientContractIT у documents-service. Видалено застарілий EnvClientContractIT, що більше не відповідав актуальному контракту.
Чому це важливо: Дотримання REST-семантики забезпечує передбачуваність API, коректну роботу HTTP-кешування та сумісність із проксі-серверами.

7. Docker Compose

Було: Базовий compose без health checks
Стало: Health-based compose з повним стеком
Як це працює: Використовується дворівневий overlay: docker-compose.yml визначає інфраструктуру (PostgreSQL, Redis, RabbitMQ, MinIO, Mailpit, Zipkin, Prometheus, Grafana), а docker-compose.services.yml — мікросервіси. Залежності використовують condition: service_healthy замість service_started, що гарантує фактичну готовність залежного компонента. JWT-секрети монтуються через volume mounts для identity-service. Файл .env.example документує усі змінні оточення. Telegram-профіль (--profile telegram) додає ngrok + webhook-init + telegram-bot-service.
Чому це важливо: Health-based залежності виключають стартові гонки, коли сервіс намагається підключитися до ще не готової інфраструктури. Overlay-підхід дозволяє запускати лише потрібні компоненти.

8. Telegram-інтеграція

Було: Відсутня
Стало: Повноцінний Telegram Bot сервіс
Як це працює: Spring Boot сервіс використовує webhook-архітектуру (не polling) — Telegram надсилає HTTP POST на ендпоінт сервісу при кожному повідомленні від користувача. Для локальної розробки ngrok створює публічний тунель, через який Telegram досягає локальний сервер. Inline-клавіатури забезпечують інтерактивний інтерфейс безпосередньо у месенджері. Сервіс комунікує з environment-service через Feign-клієнт для CRUD-операцій з feature flags. Підписка на RabbitMQ забезпечує отримання сповіщень про зміни прапорців. Авторизація на основі chat ID, per-user rate limiting, 3 Flyway-міграції, i18n: українська (uk) та англійська (en).
Чому це важливо: Telegram-бот надає оперативний доступ до управління feature flags без необхідності відкривати адмін-панель або мати доступ до серверної інфраструктури.

9. Моніторинг

Було: Відсутній
Стало: Prometheus + Grafana + Zipkin
Як це працює: Спільна бібліотека sc-observability-starter автоматично конфігурує Micrometer для експорту метрик до Prometheus у кожному сервісі, що її підключає. Grafana має 4 автоматично провізійовані дашборди (services overview, PostgreSQL, RabbitMQ, feature flags) через JSON-файли у infra/grafana/dashboards/. Zipkin забезпечує розподілене трасування — кожен HTTP-запит отримує унікальний trace ID, що дозволяє відстежити його шлях через усі мікросервіси. Усе провізіонується автоматично при першому docker compose up.
Чому це важливо: Моніторинг дозволяє виявляти проблеми продуктивності, деградацію здоров'я сервісів та міжсервісні затримки до того, як вони вплинуть на користувачів.

10. Контрактні тести

Було: Відсутні
Стало: Spring Cloud Contract (provider + consumer)
Як це працює: Provider-side тести (ProviderContractIT) визначають контракти через анотації та автоматично генерують HTTP-стаби для кожного ендпоінту. Consumer-side тести (IdentityClientContractIT) перевіряють, що Feign-клієнти відповідають контракту провайдера — тобто формат запитів та відповідей збігається. Тести розташовані у src/integrationTest/**/contract/ та запускаються як частина ./gradlew integrationTest. При зміні API-контракту тести автоматично виявляють розбіжності.
Чому це важливо: Контрактні тести ловлять ламаючі зміни API до деплою, коли виправлення ще дешеве, замість виявлення проблем у production.

11. E2E тести

Було: Відсутні
Стало: Playwright у окремому репозиторії
Як це працює: Спочатку E2E тести знаходилися у монорепозиторії, але у v1.2.0 їх мігровано до окремого репозиторію selection-committee-e2e-tests для ізоляції та спрощення CI. Тести використовують Page Object Model (POM) — кожна сторінка представлена класом з методами для взаємодії. Покриття включає: управління середовищем, доставку сповіщень, негативні сценарії. Для виконання потрібен повний Docker-стек. Доступний запис відео для дебагу.
Чому це важливо: E2E тести верифікують повний користувацький потік від UI до бази даних, виявляючи проблеми інтеграції, які не видно на рівні unit- або contract-тестів.

12. OpenAPI генерація

Було: Відсутня
Стало: TypeScript client generation
Як це працює: Директорія tools/openapi-gen/ містить config.yml та скрипти генерації (PowerShell + Bash). Скрипти зчитують OpenAPI-специфікації із запущених сервісів (ендпоінт /v3/api-docs) та генерують TypeScript-типи й код API-клієнта. Згенеровані типи забезпечують типобезпечну інтеграцію фронтенду з бекендом: будь-яка зміна контракту виявляється на етапі компіляції TypeScript, а не під час виконання.
Чому це важливо: Автоматична генерація виключає ручне дублювання типів та гарантує синхронність контракту між frontend та backend.

13. Frontend

Було: Відсутній
Стало: Next.js 16 з повним redesign
Як це працює: Застосунок побудований на App Router з розподілом на server та client компоненти. Бібліотека shadcn/ui з кастомною lavender-темою на Tailwind CSS v4 забезпечує консистентний дизайн. TanStack Query управляє серверним станом (кешування API-відповідей, автоматичний рефетчинг). Zustand використовується для клієнтського стану. React Hook Form + Zod забезпечують валідацію форм зі схемами. Biome замінює ESLint + Prettier як єдиний інструмент лінтингу та форматування. Split-panel layout для онбордінгу. Security headers (X-Frame-Options, CSP, X-Content-Type-Options) налаштовані у Next.js middleware.
Чому це важливо: Сучасний стек забезпечує продуктивність розробки, типобезпеку, консистентний UI та захист від поширених web-вразливостей.

14. Змінні оточення

Було: RABBIT_* (нестандартний префікс)
Стало: RABBITMQ_* (стандартний префікс)
Як це працює: Префікс змінних оточення стандартизовано для узгодженості з офіційним найменуванням RabbitMQ. Створено файл infra/.env.example як єдине джерело істини для всіх змінних оточення системи. Файл документує кожну змінну з очікуваними значеннями та коментарями. Перед першим запуском .env.example копіюється у .env та заповнюється реальними значеннями.
Чому це важливо: Стандартне найменування спрощує онбордінг нових розробників та зменшує ризик помилок конфігурації.

15. Інженерна документація
